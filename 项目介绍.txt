档案数字叙事AI系统（V2）——项目介绍
============================================
一、项目概述
------------

本项目名为"档案数字叙事AI系统"（Archive Narrative AI），是一个面向档案专题的智能化数字叙事生成平台。用户只需输入一个档案主题（如"焦裕禄生平事迹""C919国产大飞机""张桂梅"等），系统即可自动完成从网络资料爬取、内容分析、叙事结构规划到网页和3D虚拟展厅生成的全流程，最终输出一套完整的数字叙事作品。

项目的核心理念是：让档案"活"起来。传统的档案展示往往停留在文字罗列层面，缺乏沉浸感和叙事性。本系统利用大语言模型的规划能力、多源网络爬虫的数据采集能力、视觉模型的图片筛选能力，以及3D渲染技术的展示能力，将零散的档案资料自动编织成有结构、有温度、有视觉冲击力的数字叙事体验。


二、系统架构
------------

系统采用前后端分离的B/S架构，后端基于Python FastAPI框架，前端为单页应用。

### 2.1 后端架构（Python + FastAPI）

后端是整个系统的核心，采用"智能编排器"（SmartOrchestrator）作为唯一调度入口，将多个Agent按工作流串联执行。整体分为以下层次：

（1）API层（api/main.py）
    - FastAPI应用，监听8090端口
    - 提供RESTful API接口，包括：
      · POST /api/v2/narrative/smart —— 提交叙事生成任务（异步，返回task_id）
      · GET /api/v2/narrative/status/{task_id} —— 轮询任务状态
      · POST /api/v2/chat —— AI对话接口（支持chat和parse两种模式）
      · GET /api/v1/system/diagnostics —— 系统诊断
    - 异步任务机制：提交后立即返回task_id，前端轮询进度，避免HTTP长连接超时
    - 心跳机制：每15秒更新任务状态，前端可实时看到"正在爬取网络资料并生成作品..."等进度提示

（2）智能编排层（core/smart_orchestrator.py）
    - SmartOrchestrator是整个系统的"大脑"，负责：
      · 调用PlannerAgent规划工作流（决定执行哪些Agent、以什么顺序）
      · 按工作流依次执行各Agent，传递中间结果
      · 处理Agent失败时的重规划（replan）
      · 收集执行日志，供前端实时展示
    - 工作流典型步骤：WebCrawlerAgent → SmartAnalysisAgent → SmartOutputAgent（含WebGalleryAgent + Exhibition3DAgent）

（3）Agent层——ReAct多智能体架构（agents/ + core/react_agent.py）

    本系统的核心架构模式是"基于ReAct的多智能体系统"（ReAct-based Multi-Agent System）。
    这是一种将"推理-行动"循环与多Agent协作相结合的架构，分为三个关键层次：

    【第一层：ReAct推理引擎（core/react_agent.py）】

    ReAct（Reasoning + Acting）是一种让大语言模型交替进行"思考"和"行动"的推理范式。
    本系统实现了完整的ReActAgent基类，每个Agent都继承自它，遵循以下循环：

        思考(Thought) → 行动(Action) → 观察(Observation) → 思考 → ... → 回答(Answer)

    具体来说：
    - 思考(Thought)：LLM分析当前情况，推理下一步该做什么
    - 行动(Action)：LLM选择调用一个工具（如搜索引擎、文件解析器等），传入参数
    - 观察(Observation)：获取工具返回的结果
    - 重复以上步骤，直到LLM判断信息足够，输出最终"回答"

    ReActAgent基类的核心组件：
    - ToolRegistry（工具注册表）：每个Agent注册自己可用的工具，LLM在行动阶段选择调用
    - AgentMemory（智能体记忆）：短期记忆，记录思考/行动/观察序列，供LLM参考历史推理过程
    - MessageBus（消息总线）：Agent之间的异步通信机制，支持任务请求、结果返回、查询/应答
    - 最大迭代次数限制（默认15轮），防止无限循环

    这种ReAct模式的优势在于：
    · LLM不是一次性生成答案，而是可以"边想边做"，逐步收集信息
    · 每一步推理都可追溯、可解释，不是黑盒
    · 工具调用失败时，LLM可以换一种策略重试，具有自修复能力

    【第二层：多Agent协作（agents/）】

    系统包含多个专业化的Agent，每个Agent继承ReActAgent，拥有独立的工具集、记忆和推理能力：

    · 输入层Agent（agents/input/）
      - SmartInputAgent：智能输入处理，提取关键词、时间锚点
      - DocumentParserAgent：解析用户上传的文档（PDF/Word/TXT等）
      - WebCrawlerAgent：网络爬虫（已重构为独立爬虫模块）

    · 编排层Agent（agents/orchestrator/）
      - PlannerAgent：继承ReActAgent，利用ReAct推理循环动态规划工作流。
        它的"思考"过程是分析用户需求，"行动"是选择要调用的Agent和执行顺序，
        "观察"是评估规划结果是否合理。它不是按固定模板生成工作流，
        而是根据每次输入的不同，推理出最优的Agent调用序列。

    · 处理层Agent（agents/processing/）
      - SmartAnalysisAgent：档案内容智能分析，提取时间线、人物、事件、精神内涵

    · 输出层Agent（agents/output/）
      - SmartOutputAgent：输出编排器，协调WebGalleryAgent和Exhibition3DAgent
      - WebGalleryAgent：生成HTML叙事网页
      - Exhibition3DAgent：生成3D虚拟展厅（基于Three.js）

    【第三层：智能编排器（core/smart_orchestrator.py）】

    SmartOrchestrator是整个多Agent系统的"总指挥"，它本身不继承ReActAgent，
    而是作为上层调度器，协调各Agent的执行：

    1. 调用PlannerAgent（通过ReAct推理）生成工作流计划
    2. 按计划依次执行各Agent，通过intermediate_results在Agent间传递数据
    3. 每个Agent的输出作为下一个Agent的输入（依赖关系由PlannerAgent指定）
    4. Agent执行失败时，触发replan——重新调用PlannerAgent调整工作流
    5. 收集执行日志，供前端实时展示

    典型工作流：WebCrawlerAgent → SmartAnalysisAgent → SmartOutputAgent

    【架构总结：三层ReAct多智能体系统】

    ┌─────────────────────────────────────────────────┐
    │           SmartOrchestrator（总编排器）            │
    │   调用PlannerAgent规划 → 按序执行Agent → 失败replan │
    └──────────────────────┬──────────────────────────┘
                           │ 调度
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
    ┌────────────┐  ┌────────────┐  ┌────────────┐
    │ Crawler    │  │ Analysis   │  │ Output     │
    │ Agent      │→ │ Agent      │→ │ Agent      │
    │ (ReAct)    │  │ (ReAct)    │  │ (ReAct)    │
    └─────┬──────┘  └─────┬──────┘  └─────┬──────┘
          │               │               │
    ┌─────┴──────┐  ┌─────┴──────┐  ┌─────┴──────┐
    │ToolRegistry│  │ToolRegistry│  │ToolRegistry│
    │Memory      │  │Memory      │  │Memory      │
    │MessageBus  │  │MessageBus  │  │MessageBus  │
    └────────────┘  └────────────┘  └────────────┘

    每个Agent内部运行ReAct循环（思考→行动→观察→...→回答），
    Agent之间通过SmartOrchestrator传递中间结果，
    PlannerAgent自身也通过ReAct推理来动态规划工作流。

    这使得系统具有三个层次的智能：
    · Agent内：ReAct推理循环（微观智能）
    · Agent间：数据依赖传递（协作智能）
    · 系统层：动态规划+失败重规划（宏观智能）

（4）爬虫模块（crawler/）
    独立的爬虫子系统，采用适配器模式，支持多种搜索和数据源：

    · crawl_orchestrator.py：爬虫编排器，按搜索类型分组并行执行
    · adapters/baidu_search_adapter.py：百度网页搜索适配器（Playwright渲染SERP + 深度爬取文章）
    · adapters/baidu_image_adapter.py：百度图片搜索适配器（提取图片及来源页URL）
    · adapters/direct_adapter.py：直链页面适配器（aiohttp + Playwright回退）
    · adapters/bing_adapter.py：Bing搜索适配器
    · adapters/bilibili_adapter.py：B站搜索适配器
    · session_manager.py：HTTP会话管理
    · content_extractor.py：网页内容提取（去噪、正文识别）

（5）核心服务层（core/）
    · llm_client.py / llm_router：多LLM路由，支持qwen3-max、qwen-plus等模型的自动切换和降级
    · vision_service.py：视觉模型服务，基于qwen3-vl-max，提供图片验证、分类、分析功能
    · base_agent.py / react_agent.py：Agent基类和ReAct模式实现
    · archive_models.py：档案数据模型定义
    · bootstrap.py：系统启动和初始化

（6）工具层（utils/）
    · crawl_helpers.py：查询工程（engineer_queries）、爬取目标构建、图片候选收集、上下文补充（enrich_image_contexts）等核心工具函数
    · html_builder.py：HTML叙事网页生成器，支持5套专业配色风格、响应式设计、时间线图片嵌入
    · image_downloader.py：图片下载器，支持并发下载、VL验证、尺寸过滤、内容去重

### 2.2 前端架构（HTML + CSS + JavaScript）

前端为单页应用（web/index.html），功能包括：

    · AI对话界面：与"叙叙"（档案顾问Agent）对话，获取档案咨询
    · 生成控制面板：输入档案主题、选择输出格式（网页/3D展厅）、开关爬虫
    · 实时进度展示：工作流可视化、Agent执行日志、进度条
    · 作品画廊：浏览已生成的网页和3D展厅
    · AI自动提取关键词：输入主题后自动调用LLM解析，智能填充关键词和专题名称
    · 系统状态监控：Agent列表、LLM状态、API连接检测

### 2.3 3D展厅（Three.js）

3D展厅由Exhibition3DAgent动态生成，基于Three.js渲染：

    · 展厅结构：30m×40m的矩形展厅，含四面墙壁、天花板、地板
    · 正前方墙：序言大墙板（深色背景，展示档案主题和序言文字）
    · 左墙：时间线展板（每个事件独立墙板，含日期、标题、描述、配图）
    · 右墙：人物档案展柜
    · 后墙：精神总结墙板
    · 中央走廊：照片展柜
    · 交互方式：第一人称漫游（WASD移动 + 鼠标视角），点击展品弹出详情模态框
    · 视觉效果：阴影、雾效、ACES色调映射、环境光遮蔽


三、核心功能流程
----------------

用户输入档案主题后，系统自动执行以下流程：

第一步：智能输入解析
    - SmartInputAgent提取关键词、时间锚点、档案类型
    - 前端AI自动提取关键词功能（mode=parse模式，直接透传prompt给LLM）

第二步：查询工程与爬取目标构建
    - engineer_queries()根据档案名称和关键词，自动生成多源搜索查询：
      · 3条百度网页搜索（含百科、专题、深度）
      · 3条百度资讯搜索（新闻、专题报道、纪念活动）
      · 3条百度图片搜索（配图、历史照片、纪实照片）
      · 1条百度百科直链
      · 共计18个搜索目标
    - build_crawl_targets_from_queries()将查询转化为标准化的爬取目标

第三步：多源网络爬取
    - 爬虫编排器按搜索类型分组并行执行（网页搜索组、图片搜索组、资讯搜索组并行，组内串行防反爬）
    - 百度搜索适配器使用Playwright渲染SERP页面，提取搜索结果链接
    - 深度爬取：对每条搜索结果，并发爬取5-8篇真实文章全文（Semaphore=3控制并发）
    - 图片搜索适配器提取图片URL及data-fromurl（图片来源页URL）
    - 资讯搜索适配器爬取新闻类文章

第四步：图片智能处理（双管齐下）
    方法1——网页上下文关联图片：
    - enrich_image_contexts()为缺少上下文的图片爬取来源页，提取图片周围文字
    - 百度图片搜索提取data-fromurl，获取图片在原网页中的上下文
    - 并发爬取来源页（Semaphore=4），用extract_images_with_context提取上下文

    方法2——视觉模型识别图片内容：
    - VisionService基于qwen3-vl-max，对每张图片进行验证和分类
    - 验证维度：相关性（relevant）、质量评分（quality_score）、历史价值（historical_score）
    - 综合评分：composite_score = 相关性×0.5 + 质量×0.25 + 历史价值×0.25
    - 有充分上下文（≥20字）的图片跳过VL验证，节省API调用
    - 上下文辅助通过：有上下文且综合分≥0.35时，即使VL判定不相关也给予通过

第五步：档案内容智能分析
    - SmartAnalysisAgent基于爬取的文本和图片，利用LLM提取：
      · 时间线事件（日期、标题、描述）
      · 人物档案（姓名、角色、简介）
      · 核心精神与价值
      · 叙事结构建议
    - 生成结构化的叙事框架，供输出Agent使用

第六步：多格式输出生成
    - HTML叙事网页（WebGalleryAgent + html_builder.py）：
      · 序言 → 总览 → 时间线 → 图片展示 → 总结
      · 5套专业配色风格（根据主题自动选择）
      · 响应式设计，适配手机/平板/桌面
      · 时间线图片嵌入，图文对照

    - 3D虚拟展厅（Exhibition3DAgent + exhibition_3d.js模板）：
      · 动态生成展厅布局和展品
      · 墙板文字纹理（Canvas动态渲染）
      · 图片匹配算法：上下文n-gram匹配权重加倍
      · 第一人称漫游交互


四、技术亮点与特色
------------------

### 4.1 智能编排与自适应工作流

系统没有采用硬编码的流水线，而是由PlannerAgent根据输入动态规划工作流。当某个Agent执行失败时，编排器会调用replan重新规划，跳过失败步骤或尝试替代方案。这种自适应能力使系统具有较好的鲁棒性。

### 4.2 查询工程（Query Engineering）

这是本项目的一个核心创新点。系统不是简单地用档案名称去搜索，而是通过engineer_queries()函数，根据档案名称、关键词、时间锚点，自动生成多维度、多类型的搜索查询：
    - 网页搜索覆盖百科知识、专题报道、深度解读
    - 资讯搜索覆盖最新新闻、纪念活动
    - 图片搜索覆盖配图、历史照片、纪实照片
    - 每种搜索类型使用不同的百度搜索参数（如资讯搜索使用tn=news参数）

这种"查询工程"思路确保了数据采集的广度和深度，是系统能生成高质量叙事内容的基础。

### 4.3 图片匹配的"双管齐下"策略

图片与文字内容的精准匹配是数字叙事的关键难题。本项目创新性地采用了双管齐下策略：

    - 方法1（网页上下文关联）：利用图片在原网页中的上下文文字确定图片与什么内容相关。百度图片搜索的图片通常缺少上下文，但带有来源页URL，系统爬取来源页提取上下文。
    - 方法2（视觉模型识别）：使用qwen3-vl-max视觉大模型，对图片内容进行语义理解和分类判断。

在最终匹配时，上下文匹配权重（4）> VL描述权重（3）> 路径匹配权重（1），确保有上下文的图片优先匹配到正确位置。

### 4.4 爬虫系统的适配器模式与并行优化

爬虫模块采用适配器模式，每种搜索源对应一个适配器，新增数据源只需添加适配器即可。在执行策略上：
    - 按搜索类型分组并行（网页搜索、图片搜索、资讯搜索同时进行）
    - 组内串行执行并加入防反爬延迟
    - 深度爬取文章使用Semaphore(3)并发控制
    - 上下文补充使用Semaphore(4)并发控制

这种"分组并行+组内串行"的策略，在保证不被反爬的同时最大化了爬取效率。

### 4.5 VL验证的智能跳过机制

图片VL验证是系统中最耗时的环节之一（每张图片一次API调用）。系统实现了智能跳过：当图片已有20字以上的上下文文字时，说明该图片在原网页中就有明确的语义关联，跳过VL验证直接通过。这一优化在保证质量的前提下，将VL验证的API调用量减少了约40-60%。

### 4.6 3D展厅的动态生成

3D展厅不是预设模板，而是Exhibition3DAgent根据分析结果动态生成的。展厅的墙板文字使用Canvas动态渲染为纹理贴图，时间线事件、人物档案、精神总结等展品都根据实际内容动态放置。这意味着不同的档案主题会生成完全不同的展厅布局和内容。

### 4.7 前端轮询容错机制

前端任务轮询采用容错设计：单次fetch失败不会导致整个任务报错，而是跳过继续轮询；连续5次失败才判定为任务失败。这避免了因网络抖动导致的误报。

### 4.8 多LLM路由与降级

系统支持多个LLM模型的自动切换和降级。当主模型（qwen3-max）调用失败时，自动降级到备用模型（qwen-plus），确保系统可用性。

八、技术栈总结
--------------

    后端框架：Python 3.11 + FastAPI
    大语言模型：Qwen3-max / Qwen-plus（通义千问）
    视觉模型：Qwen3-VL-max
    3D渲染：Three.js（WebGL）
    网页爬虫：Playwright + aiohttp + BeautifulSoup
    前端：原生HTML/CSS/JavaScript（无框架依赖）
    异步编程：asyncio + Semaphore并发控制
    数据验证：Pydantic v2
